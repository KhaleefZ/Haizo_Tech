React Server Components, one year in: where we use them and where we don't

An honest account after shipping three apps on the App Router. Server Components are genuinely better for data-heavy pages that mostly read. They are a poor trade for anything with dense client interaction, and most of the pain people report comes from picking one answer for the whole application.

Khaleefulla Z
Co-founder · 19 May 2026 · 13 min read

The framing that caused the most trouble in the first few months was treating Server Components as a replacement for client rendering. They are not a replacement. They are a second place to run component code, with different capabilities, and the interesting engineering is in deciding where the line between the two falls.

Get that line in the right place and a page ships less JavaScript and fetches data without a round trip. Get it wrong and you spend a week discovering that a context provider three levels up has quietly turned your entire route into a client bundle.

What actually changes

A Server Component runs once, on the server, and never ships to the browser. Its code is not in the bundle. It can read from the database directly, use secrets, and await in the component body. What it cannot do is anything stateful or interactive: no useState, no useEffect, no event handlers, no browser APIs.

A Client Component is the React you already know. It ships to the browser, hydrates, and handles interaction. The 'use client' directive marks the boundary — and this is the part that trips people up, because the directive is not a per-component setting.

The boundary is a subtree, not a file

Marking a module 'use client' puts that module and everything it imports into the client graph. A single directive at the top of a layout can pull the whole route into the browser bundle, and nothing warns you, because everything still works. It just works the way it did before Server Components existed. 

// Wrong. The whole page is now a Client Component because one
// button needed onClick — so BookingTable, its formatting helpers
// and the date library it imports all ship to the browser.
'use client';

export default async function Page() { /* ... */ }


// Right. The page stays on the server. Only the interactive leaf
// crosses the boundary, and it receives already-rendered children.
import { ExportButton } from './export-button'; // 'use client'

export default async function Page() {
  const bookings = await db.bookings.upcoming();
  return (
    <section>
      <BookingTable rows={bookings} />   {/* server */}
      <ExportButton count={bookings.length} /> {/* client */}
    </section>
  );
}

The rule that follows from this: push 'use client' as far down the tree as it will go. Interactive leaves, not interactive pages. A Server Component can render a Client Component, and a Client Component can render Server Components passed to it as children — that second direction is the escape hatch that makes deeply interactive layouts workable, and it is worth learning early.

Where they clearly win

The pattern where Server Components are unambiguously the better tool is a page that reads a lot of data, renders it, and offers only shallow interaction. Dashboards, list and detail views, reports, content pages, admin tables.

  • The data fetch has no round trip. The component queries the database in the same process. No API call from the browser, no loading spinner for the initial view, no waterfall where the page must render before it knows what to request.
  • Heavy dependencies stay on the server. Markdown renderers, date and currency formatters with full locale data, syntax highlighters, validation schemas — all of it runs where it costs nothing in bundle size.
  • Secrets stay put. There is no client bundle to accidentally leak an API key into, because the code never leaves the server.
  • Streaming gives you real progressive rendering. Wrap a slow section in <Suspense> and the rest of the page reaches the user immediately. The fallback is server-rendered too, so it is not a blank frame.
  • Less state to manage. A large amount of client state exists purely to cache server data. When the server renders it, that machinery disappears along with it.

Where they are a poor trade

The other side is just as clear once you have hit it. If a screen's core loop is interaction — dragging, filtering, editing in place, live validation, optimistic updates — you will end up marking nearly everything 'use client' anyway, and you will have paid the framework's complexity for none of its benefit.

The tells

  • Most components need state. If nine out of ten leaves need useState, the boundary is not buying you a smaller bundle. It is just an extra concept.
  • Interaction must feel instant. Anything that round-trips to the server to re-render has a floor set by network latency. For a filter that should respond as the user types, that floor is too high — filter on the client over data you already have.
  • The state is genuinely client-owned. A half-completed multi-step form, a canvas, an editor, a drag operation in progress. None of that belongs on the server, and trying to put it there produces worse code than the client-only version.
  • You are fighting serialisation. Props crossing the boundary must be serialisable. Passing functions, class instances or Date-heavy structures around gets awkward fast, and the awkwardness is a signal the split is in the wrong place.
The decision is per route, not per project. This is the single most useful thing we learned. The same application can have a Server-Component-first reporting section and a thoroughly client-rendered scheduling screen, and both can be the right call. Teams that suffer with the App Router are usually the ones that adopted a rule for the whole codebase and then defended it.

The parts that stayed genuinely awkward

Caching is the real learning curve

Server Components themselves are simple. Deciding what is static, what revalidates on a timer, what revalidates on a tag, and what must never be cached is not — and the defaults have shifted between framework versions often enough that most answers you find online are for a different version than the one you are running. Be explicit about caching at every fetch rather than relying on defaults, and write down which routes are dynamic and why.

Error boundaries and loading states need designing

Streaming means parts of a page can fail independently. That is a feature, but it means "what does this page look like when one section errors" is now a real design question with a real answer required for each section, not a single top-level error screen. The framework gives you the mechanism; it does not give you the decision.

Third-party components assume the client

A great many libraries are written with the implicit assumption that they run in a browser. Most work fine once wrapped in a small client component, but "wrap it" is a step you take repeatedly, and each wrapper is a place the boundary can creep upward if you are not watching.

What to take away

Server Components are worth using, and the debate about whether they are good is less useful than knowing precisely what they are good at. Our working rules:

  • Default to server rendering for anything that mostly reads and displays. Default to client rendering for anything whose primary purpose is interaction.
  • Decide per route. Mixed applications are normal and fine.
  • Keep 'use client' on leaves. Audit the bundle occasionally to catch the directive that migrated upward during a refactor.
  • Be explicit about caching. Never inherit a default you cannot state out loud.
  • If a screen ends up almost entirely client components, that is not a failure — it is the correct answer for that screen, and you should stop trying to fix it.

The technology is not the hard part. Resisting the urge to apply one answer everywhere is.
