The index you didn't add is the query you can't explain

A booking availability query went from 40ms to 2.3 seconds over eighteen months without a single code change. Nothing was refactored, no dependency was upgraded, no feature was added. The table simply got bigger, and the plan Postgres had been choosing stopped being the right one.

GS
Gokul S
Co-founder · 24 June 2026 · 11 min read

Query performance is not a property of a query. It is a property of a query and the data it runs against. The planner picks a strategy using statistics about what is in the table right now, so the same SQL can be answered by a completely different plan six months later. That is not a bug. It is the planner doing its job with different inputs.

The practical consequence is that slow queries usually do not need to be rewritten. They need to be understood. Below is the sequence we work through, in order, before anyone is allowed to add an index.

Read the plan before touching the schema

The most common mistake is to guess. Someone sees a slow query on a table with a status column and a created_at column, adds an index on both, and moves on. Sometimes it helps. Often it does nothing, and now there is an extra index to maintain on every write.

EXPLAIN tells you the plan; ANALYZE tells you the truth

Plain EXPLAIN shows the plan the planner intends to use and its estimated row counts. EXPLAIN (ANALYZE, BUFFERS) actually runs the query and reports what really happened, including how many blocks came from cache versus disk. The gap between estimated and actual rows is the single most useful number on the page: when the planner thinks a step will return 40 rows and it returns 90,000, every decision built on top of that estimate is wrong. 

-- Suppose bookings has ~2 million rows, of which a few thousand are 'held'.
EXPLAIN (ANALYZE, BUFFERS)
SELECT id, room_id, starts_at
FROM   bookings
WHERE  status = 'held'
AND    starts_at >= now()
ORDER  BY starts_at
LIMIT  50;

Limit  (cost=0.43..812.10 rows=50 width=28)
       (actual time=1904.221..2287.664 rows=50 loops=1)
  Buffers: shared hit=118204 read=39117
  ->  Index Scan using bookings_starts_at_idx on bookings
        (cost=0.43..71422.88 rows=4398 width=28)
        (actual time=1904.219..2287.602 rows=50 loops=1)
        Index Cond: (starts_at >= now())
        Filter: (status = 'held'::text)
        Rows Removed by Filter: 1443902
Planning Time: 0.214 ms
Execution Time: 2287.719 ms

The Rows Removed by Filter line is the whole story. Postgres is walking the starts_at index in order — which is genuinely the cheapest way to satisfy ORDER BY starts_at LIMIT 50 — and throwing away nearly one and a half million rows on the way, because almost none of them have the status we want. At launch, when the table held a few thousand future bookings, it found fifty matches almost immediately. The plan never changed. The distance it had to walk did.

Why the obvious composite index made it worse

The reflex fix is CREATE INDEX ON bookings (starts_at, status). It reads like it covers the query: both columns are there, in the order they appear in the WHERE clause. It does not help at all.

Column order is not cosmetic

A B-tree index is sorted by its first column, then by its second within each value of the first, and so on. That structure determines what it can do:

  • An equality predicate on a leading column narrows the scan to a contiguous slice of the index. This is the cheap case.
  • A range predicate on a leading column also narrows to a contiguous slice — but every column after it is now unordered within that slice, so it can only be used as a filter, not to seek.
  • Because of that, put equality columns first and range columns last. (status, starts_at) is a different index from (starts_at, status), and only one of them is useful here.
  • The index's sort order can also satisfy ORDER BY for free, but only if the ordering columns are a prefix of the index — or follow equality-constrained columns.

With (starts_at, status), Postgres still walks the same range of the index in the same order and still discards the same rows. It has just made the index physically larger, so each page holds fewer entries and the walk touches more blocks. Measurably worse, for exactly the reason it looked correct.

Flipping to (status, starts_at) is the right shape: seek directly to the held section, then read it in starts_at order and stop after fifty rows. No filtering, no sort.

The partial index that actually fixed it

(status, starts_at) works, but it indexes all two million rows, including every cancelled and completed booking that no availability query will ever ask about. Those rows cost write throughput and disk, and they buy nothing.

A partial index stores only the rows matching a predicate: 

CREATE INDEX CONCURRENTLY bookings_held_starts_at_idx
    ON bookings (starts_at)
 WHERE status = 'held';

-- The planner uses a partial index only when it can prove the query's
-- WHERE clause implies the index predicate. `status = 'held'` does.
-- `status IN ('held','confirmed')` does not — that needs its own index
-- or a wider predicate.

Note that status does not need to be a column in the index — it is in the predicate instead. Every entry in this index is already known to be held, so the index only has to store starts_at. On a table where held bookings are a small fraction of the rows, that is a dramatically smaller structure: faster to scan, cheaper to keep in cache, and untouched by writes to rows that do not match the predicate.

CONCURRENTLY matters in production. Without it, CREATE INDEX takes a lock that blocks writes to the table for the duration of the build. With it, the build takes two passes and longer overall, but the table stays writable. It cannot run inside a transaction block, and if it fails it leaves an invalid index behind that you have to drop before retrying — both worth knowing before you run it at 9am.

The trap in partial indexes: they are only as good as the assumption behind the predicate. Index WHERE status = 'held' and then add a pending_payment status that availability also needs to consider, and the planner silently stops using the index for the new query. It does not warn you. It just goes back to the plan you had before, and you are debugging the same slowdown a year later. Partial indexes belong next to a comment explaining what they assume.

Statistics are the other half of the problem

Sometimes the index exists and the planner still refuses to use it, because its row estimate is badly wrong. Two things to check before blaming the planner:

  • Is autovacuum keeping up? Statistics are refreshed by ANALYZE, which autovacuum runs on its own schedule. On a table with heavy churn, the defaults can lag far enough that the planner is working from a snapshot of a much smaller table.
  • Is the default statistics target high enough? Postgres samples a fixed number of values per column to build its histogram. On a column with a very skewed distribution, raising default_statistics_target for that column and re-running ANALYZE can change the plan without changing the schema.
  • Are the columns correlated? Postgres assumes independence between predicates by default. If status and starts_at are related — and in a booking system they very much are, since old bookings are rarely still held — the estimate for the combination will be off. Extended statistics via CREATE STATISTICS tell the planner about that correlation.

What to take away

The useful discipline is not "add indexes for slow queries". It is closer to this:

  • Slow queries that were once fast are usually a distribution change, not a code change. Look at how much data the plan is discarding before you look at the SQL.
  • Rows Removed by Filter and the estimated-versus-actual row gap tell you more than the total execution time does.
  • Equality columns before range columns. (a, b) and (b, a) are not interchangeable, and picking the wrong one can be slower than having no index.
  • When a query only ever cares about a small, stable subset of rows, a partial index is smaller, faster, and cheaper to write than a full one — as long as the predicate is documented and revisited when the states change.
  • Build indexes CONCURRENTLY on live tables, and know the failure mode before you run it.

The query in the title never got rewritten. It is the same SQL it was on day one. The difference is that we now know why it is fast, which means we will notice when that stops being true.
