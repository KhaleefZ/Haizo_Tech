Writing the evaluation set before writing the RAG pipeline

Fifty real questions with known answers, collected from the client's own inbox, turned an unfalsifiable demo into something we could improve on purpose. The eval set is not a testing chore you do at the end. It is the specification, and writing it first changes what you build.

GS
Gokul S
Co-founder · 17 March 2026 · 12 min read

A retrieval-augmented generation demo is unusually easy to build and unusually hard to improve. You can have documents chunked, embedded and answering questions in an afternoon, and it will feel impressive, because you will type questions you already know it can answer.

Then someone asks whether a smaller chunk size would be better, or whether reranking is worth the latency, or whether last week's change helped — and there is no way to answer. The only instrument is opening the demo and typing a few things, which measures your mood more than the system. Every change is a coin flip you cannot score.

The fix is not a better model or a better vector store. It is having something to measure against, built before the thing being measured.

What an eval set actually is

A list of questions, each paired with what a correct answer requires and which source documents contain it. That is all. It does not need tooling, a platform or a framework — a spreadsheet that becomes a JSON file is entirely adequate to start. 

{
  "id": "refund-window-partial",
  "question": "we cancelled 3 of 5 seats, do we get anything back?",

  // What a correct answer must contain. Not a golden string —
  // there are many correct phrasings and only these facts matter.
  "must_include": [
    "partial cancellations are refunded pro rata",
    "only within 14 days of booking"
  ],

  // What it must NOT do. Usually the more useful field.
  "must_not_include": [
    "a specific refund amount or percentage"
  ],

  // Which chunks SHOULD have been retrieved. This is what lets
  // you score retrieval on its own, separately from the answer.
  "expected_sources": ["policy-v4.md#refunds", "policy-v4.md#partial"],

  "category": "multi-hop"
}

Real questions, not questions you invented

This is the part that cannot be shortcut. Questions written by the people building the system are systematically unrepresentative: they are well-formed, they use the vocabulary of the documentation, and they ask about things the authors know are covered.

Real user questions are misspelled, missing context, phrased in the user's vocabulary rather than the document's, and frequently about things the documentation does not address at all. A support inbox, a chat log, or a list of the questions the team is tired of answering is worth more than a hundred invented ones. Ask the client for theirs — it costs them an export and it is the single highest-leverage input to the project.

Include the questions that should fail

An eval set of only answerable questions optimises for a system that always answers, which is exactly the failure mode people complain about. Deliberately include:

  • Out-of-scope questions the documents genuinely do not cover. The correct behaviour is to say so. If it invents an answer here, that is the most important bug you have.
  • Ambiguous questions with more than one reasonable reading. The correct behaviour is usually to ask, or to cover both.
  • Questions with outdated answers in the corpus, where an old and a new document conflict. Retrieval will happily find both.
  • Multi-hop questions needing two documents combined. These fail far more often than single-fact lookups and are invisible if you only test the easy case.
  • Near-miss questions that sound like a covered topic but differ in a material detail. These catch retrieval that is matching on topic rather than meaning.

Score retrieval separately from generation

The most useful structural decision is to measure the two stages independently, because they fail for unrelated reasons and the fixes have nothing in common.

Retrieval metrics are cheap and objective

Retrieval scoring needs no model and no judgement. Given the expected source chunks, you can compute directly:

  • Recall@k — was the needed chunk in the top k results? If it was not, no amount of prompt engineering will save the answer. This is the first number to look at, always.
  • Mean reciprocal rank — how far down was it? Position 1 and position 9 both count for recall@10, but they behave very differently once a context window fills up.
  • Precision — how much irrelevant material came along? Noisy context degrades answers and costs tokens.

Suppose your run shows recall@5 at 0.62. That single number tells you where the next week goes: nearly four in ten questions never had the right material available, and chunking, embedding choice, hybrid keyword search or reranking are the levers — not the prompt.

Generation scoring is fuzzier, and that is fine

Given correct context, did the answer contain the required facts, avoid the forbidden ones, and cite its sources? Exact string matching is too brittle for natural language, so a model-as-judge with a narrow, specific rubric works well — as long as you spot-check the judge against human ratings periodically. A judge that has drifted is worse than no judge, because it is confidently wrong at scale.

Fix retrieval before you touch the prompt. Prompt changes are fast, satisfying and highly visible, which is why teams reach for them first. But if the right chunk was never retrieved, the prompt is irrelevant — you are refining how the model phrases a guess. Recall@k tells you which problem you actually have, and it is almost always the one nobody wants to work on.

Running it as a build step

Once the set exists, wire it into the workflow the same way you would a test suite: run it on every meaningful change, keep the scores in version control, and compare against the previous run rather than against an absolute target.

Two things this catches that nothing else does. First, regressions — a change that improves five questions and breaks three is trivially detectable and completely invisible in manual testing. Second, corpus changes. When someone updates the source documents, the system's behaviour changes without a single line of code changing, and the eval set is the only thing watching that boundary.

Keep a holdout portion you do not look at during development. It is genuinely possible to overfit a pipeline to fifty questions by tuning chunk boundaries until those specific questions work, and a holdout is how you find out you have done it.

Before the eval set, every discussion about the pipeline was a matter of opinion and the loudest person won. After it, the discussion was about a number, and the argument took fifteen minutes instead of a week.

What to take away

  • Write the eval set first. It forces the questions "what is this for" and "what does correct mean" to be answered while they are still cheap to answer.
  • Use real questions from real users. Invented questions test a system nobody will use.
  • Include unanswerable, ambiguous, multi-hop and near-miss questions. A set of easy questions optimises for a system that is confidently wrong.
  • Measure retrieval and generation separately. Recall@k tells you which half of the system is failing, and it is usually not the half people want to work on.
  • Fifty good questions beat five hundred generated ones. Quality and representativeness matter far more than size at this stage.
  • Run it on every change, track the scores over time, and keep a holdout.

The pipeline itself is not the hard part — the components are well understood and largely interchangeable. Knowing whether your particular combination of them is getting better is the hard part, and that is a decision you make on day one or not at all.
