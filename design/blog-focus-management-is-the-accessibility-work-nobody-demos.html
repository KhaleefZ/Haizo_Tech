Focus management is the accessibility work nobody demos

Modals, route changes and async lists all move focus somewhere — and if you have not decided where, the browser decides for you, usually badly. Getting it right is unglamorous, nearly free while you are building the component, and painful to retrofit across a finished application.

KZ
Khaleefulla Z
Co-founder · 26 February 2026 · 7 min read

Focus is the browser's cursor: the single element that receives keystrokes, and the place a screen reader considers "here". For anyone navigating by keyboard — permanently, or temporarily because they are using a trackpad-less laptop, or because a hand is occupied — focus position is the entire interface. It is the equivalent of where a sighted mouse user is looking.

The reason it gets missed is that it is invisible to the way most software is reviewed. Nobody demos focus. A stakeholder walks through a feature with a mouse, everything looks right, and the fact that opening a dialog dropped focus to the top of the document is simply not part of what anyone observed.

The three places focus goes wrong

Opening and closing dialogs

When a dialog opens, focus must move into it. If it does not, a keyboard user is still tabbing through the page behind — which is often still interactive, and which they can no longer see clearly. A screen reader user gets no announcement that anything happened at all.

Three things have to be true for the lifetime of the dialog:

  • Focus moves in on open — to the dialog container, its heading, or the first meaningful control. Prefer the container or heading, so the accessible name is announced before the controls.
  • Focus is trapped while it is open — tabbing past the last element wraps to the first, not out into the page behind. Content behind must also be inert, so it is not reachable by any means.
  • Focus returns on close — to the element that opened it. Not the top of the page. Users close a dialog expecting to be back where they were, and being teleported to the document start means re-navigating the entire page.

The native <dialog> element does most of this for you, and is worth preferring over a hand-built one. If you are building it yourself, the return step is the one most often forgotten, because it only shows up when you actually close the thing with a keyboard. 

// Capture the opener BEFORE moving focus, restore it after.
// document.activeElement is already the dialog by the time
// most cleanup functions run, so capturing late captures nothing.
useEffect(() => {
  if (!open) return;

  const opener = document.activeElement;
  panelRef.current?.focus();

  return () => {
    // The opener may have unmounted while the dialog was open —
    // deleting a row from its own confirm dialog, for example.
    // Fall back to a stable landmark rather than losing focus.
    const target = opener?.isConnected
      ? opener
      : document.getElementById('main');
    target?.focus();
  };
}, [open]);

Note the container needs tabIndex={-1} to be focusable programmatically without being in the tab order. That single attribute is the difference between .focus() working and silently doing nothing.

Client-side route changes

A full page load resets focus to the top of the new document and screen readers announce the new page. A client-side route change does neither: the URL changes, the content changes, and focus stays exactly where it was — on a link that no longer exists, which means focus falls back to <body>.

The user's next Tab starts from the very beginning of the document. Navigate three levels deep in a client-rendered app and you have tabbed through the header three times. Nothing announces that the page changed, so a screen reader user may not know navigation happened.

The fix is to move focus deliberately on navigation — to the new page's h1, or to the main landmark, with tabIndex={-1} so it can receive focus. Pair it with a live region announcing the new page title if you want the change spoken explicitly. This is roughly ten lines in a router-level effect, and it fixes every route at once, which makes it one of the highest-value accessibility changes available in a single-page app.

Content that appears, disappears or reorders

The subtlest category. Focus is attached to a DOM element, so if that element is removed, focus is lost — silently.

  • Deleting a list item while its own delete button has focus. Focus falls to body. Move it to the next item, or to the previous one if it was last, or to the list container if it is now empty.
  • Replacing content with a loading skeleton. The focused element vanishes mid-interaction. Prefer keeping the trigger mounted and disabled over unmounting the region around it.
  • Async results arriving. Do not steal focus to new results — that interrupts someone mid-typing. Announce them in a polite live region instead, and let the user choose to move.
  • Expanding accordions and disclosures. Focus stays on the trigger, which is correct. What matters is that aria-expanded updates so the state change is announced.
  • Infinite scroll. Appending content below is fine; prepending or reordering above the focused element moves the viewport under the user.
Never remove the focus indicator. outline: none with no replacement is the most common accessibility defect on the web, and it is almost always applied deliberately because a designer found the default ring ugly. The correct response is to design a better one. Use :focus-visible so the ring appears for keyboard users without showing on mouse clicks — that gets you the clean visual people actually wanted, without removing the only way keyboard users can tell where they are.

Testing it takes five minutes

You do not need a screen reader or an audit tool for the majority of these. Put the mouse down and use the feature with the keyboard alone:

  • Can you reach every interactive element with Tab, and can you always see where you are?
  • Does Tab order follow the visual order? A mismatch usually means CSS reordered something the DOM did not.
  • Open a dialog: does focus go in, stay in, and come back to the trigger on Escape?
  • Navigate to another route: where does the next Tab start?
  • Delete an item from a list: is focus somewhere sensible afterwards?
  • Is there anything you can reach but not activate, or activate but not escape?

Automated tooling is genuinely useful and genuinely limited here. It reliably catches missing labels, contrast failures and invalid ARIA. It cannot tell you that focus landed in a reasonable place after an interaction, because "reasonable" is a judgement about intent. That part needs a person, for five minutes, per feature.

Why retrofitting is expensive

Handled while building a component, focus management is a few lines inside a component you already have open, with the interaction fresh in your mind.

Handled afterwards, it is an audit of every interactive component in the application, by someone who did not write most of them, working out what each one intended. The fixes themselves are still small — but finding them, verifying them, and regression-testing a hundred components is a different size of job entirely. And because the fixes touch focus and mounting behaviour, they land in exactly the code most likely to have subtle dependencies on the current behaviour.

What to take away

  • Focus is where the keyboard user is. Every interaction that changes the DOM should have a deliberate answer for where focus ends up.
  • Dialogs: focus in, trapped while open, returned to the opener on close. Capture the opener before you move focus.
  • Client-side navigation: move focus to the new page's heading or main landmark. One router-level fix covers the whole app.
  • Removing focused elements loses focus silently. Decide where it goes before you unmount them.
  • Keep a visible focus indicator. Style it with :focus-visible; never delete it.
  • Test with the keyboard, per feature, while building. It is five minutes then and a project later.

None of this is difficult, and none of it will ever be pointed at in a demo. It is simply the difference between an application some people can use and an application everyone can use, and it is decided by whether anyone thought about it while the component was being written.
