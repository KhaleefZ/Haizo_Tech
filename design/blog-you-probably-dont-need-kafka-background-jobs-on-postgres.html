You probably don't need Kafka: background jobs on Postgres

A correct, concurrent, durable job queue is about forty lines of SQL in a database you are already running, already backing up and already monitoring. The reason to reach for dedicated infrastructure is a specific requirement, not a vague sense that a table is not a serious enough place to keep jobs.

GS
Gokul S
Co-founder · 9 April 2026 · 9 min read

The usual objection to a database-backed queue is that multiple workers polling one table will collide: two of them read the same row, both think they own it, and the welcome email goes out twice. That objection was correct for a long time, and it is the reason a generation of developers learned that queues belong in dedicated infrastructure.

Postgres 9.5 added SKIP LOCKED, and the objection stopped being true. It is worth understanding exactly what it does, because it is the whole mechanism.

How SKIP LOCKED makes the naive queue correct

Normally, if a transaction tries to lock a row another transaction already holds, it waits. For a job queue that is the worst possible behaviour: every worker queues up behind the first one and you have accidentally built a single-threaded system with extra steps.

SKIP LOCKED tells Postgres to ignore rows it cannot immediately lock and move on to the next candidate. Each worker therefore gets a different row, with no coordination, no advisory locks, and no polling-and-retrying. 

CREATE TABLE jobs (
  id           bigserial PRIMARY KEY,
  kind         text        NOT NULL,
  payload      jsonb       NOT NULL,
  run_at       timestamptz NOT NULL DEFAULT now(),
  attempts     int         NOT NULL DEFAULT 0,
  max_attempts int         NOT NULL DEFAULT 20,
  locked_at    timestamptz,
  last_error   text
);

-- Partial index: only rows a worker could actually claim.
-- Finished jobs are deleted, so this index stays small even
-- if the table has churned through millions of rows.
CREATE INDEX jobs_ready_idx ON jobs (run_at, id)
  WHERE locked_at IS NULL;

The claim is a single statement. One round trip, atomic, and it hands back the job the worker now owns: 

UPDATE jobs
   SET locked_at = now(),
       attempts  = attempts + 1
 WHERE id = (
   SELECT id FROM jobs
    WHERE locked_at IS NULL
      AND run_at <= now()
    ORDER BY run_at, id
    FOR UPDATE SKIP LOCKED
    LIMIT 1
 )
 RETURNING id, kind, payload, attempts;

Run twenty workers against this and each gets a distinct job. Kill a worker mid-job and its transaction aborts, the lock is released, and the row becomes claimable again — durability you get from the database rather than from your own reconciliation code.

Delivery semantics, stated plainly

This gives you at-least-once delivery. A worker can complete a job's side effect and then die before marking it done, and the job will run again. That is not a flaw in the design; it is the honest guarantee of every queue that survives worker failure. Systems claiming exactly-once are almost always providing at-least-once plus deduplication somewhere.

The consequence is that handlers must be idempotent, and this is the part teams skip. Practically: use an idempotency key on outbound API calls, use INSERT … ON CONFLICT DO NOTHING for inserts, and check whether the effect has already happened before performing it. This is required with any queue technology; it is just more visible here.

Where the ceiling actually is

The honest answer is that the ceiling is high enough that most applications will never approach it, and the specific number depends entirely on your hardware, job duration, and how much work each job does in the same database.

On a modest managed instance we have seen this pattern sustain a few thousand claims per second with the claim query itself staying in single-digit milliseconds. For context: a hypothetical application sending 50,000 notifications a day averages well under one job per second. Even a very spiky workload that compresses a day into one hour is nowhere near this ceiling.

What actually degrades first, in our experience, is rarely the claim query:

  • Table bloat from completed jobs. If finished jobs are marked done rather than deleted, the table grows without bound and autovacuum falls behind. Delete on success and move failures to a separate dead_jobs table.
  • Polling cost at idle. Twenty workers polling every 100ms is 200 queries per second against an empty queue, forever. Use LISTEN/NOTIFY to wake workers on new jobs, with a slow poll as a safety net.
  • Connection count. Each worker holds a connection. This is usually the real limit on a managed Postgres, and it is a pooling problem, not a queue problem.
  • Long transactions. If a handler does its work inside the claiming transaction, a slow job holds a transaction open for its whole duration, which blocks vacuum across the database. Claim in a short transaction, then work outside it.
  • Jobs competing with user traffic. The queue shares CPU and I/O with the application. This is a genuine downside of colocating them, and the reason the answer changes at scale.
Delete completed jobs; don't mark them. This one decision accounts for most of the difference between a Postgres queue that stays fast for years and one that becomes a support burden after six months. If you need the history, write a row to an append-only job_history table and delete from jobs — a hot, small, high-churn table and a large, cold, append-only one have completely different maintenance profiles, and combining them gives you the worst of each.

When to actually reach for something else

There are real reasons to run dedicated infrastructure. Kafka in particular is not a job queue at all — it is a durable, replayable, ordered log, and the cases where you want one are specific:

  • Multiple independent consumers of the same stream. A queue consumes a job once. A log lets five unrelated services each read every event at their own pace, and lets a sixth be added next year and replay history from the beginning.
  • Retention and replay as a product requirement. If "re-process the last thirty days through a fixed consumer" must work, you need a log.
  • Throughput genuinely beyond one database. Real, measured, sustained — not projected from a growth deck.
  • Strict per-key ordering across partitions. Achievable in Postgres, but it stops being forty lines of SQL.
  • Isolation from the primary database. A legitimate reason on its own: you may simply not want batch work sharing resources with user requests.

None of those are "we might get big". If the requirement is not one you can state in a sentence today, the operational cost of a second stateful system — its own failure modes, backups, upgrades, monitoring and on-call knowledge — is being paid for a benefit that does not exist yet.

What to take away

  • FOR UPDATE SKIP LOCKED makes a table into a correct concurrent queue. The old objection about workers colliding no longer applies.
  • You get at-least-once delivery, so handlers must be idempotent. That is true of every queue; here it is at least explicit.
  • Claim in a short transaction and do the work outside it. Never hold a transaction open for the duration of a job.
  • Delete completed jobs. Index partially on claimable rows. Wake workers with NOTIFY instead of tight polling.
  • Move to dedicated infrastructure for a requirement you can name — fan-out, replay, measured throughput, isolation — not for a feeling about seriousness.

The strongest argument for the boring option is not performance. It is that jobs live in the same transaction as the data that created them. Enqueue a welcome email in the same transaction that creates the user, and it is impossible to have a user without a queued email, or an email for a user whose creation rolled back. That guarantee is free here, and expensive to reconstruct across two systems.
