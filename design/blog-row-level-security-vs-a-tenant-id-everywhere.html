Row-level security vs. a tenant_id everywhere: what we chose for SchoolOS

Two ways to isolate tenants inside one Postgres database, each with a failure mode worth understanding before you commit. We ended up enforcing isolation in two places rather than one, which sounds like indecision and is actually the point.

GS
Gokul S
Co-founder · 28 April 2026 · 10 min read

In a multi-tenant application, the worst bug you can ship is one tenant seeing another tenant's data. It is not a degraded experience; it is a breach, and in a school management system it is a breach involving children's records. So the interesting question is not which approach is more elegant. It is which approach fails safely when someone makes an ordinary mistake at 6pm on a Friday.

Assume the shared-database, shared-schema model throughout — one Postgres database, one set of tables, a tenant_id column on every tenant-scoped row. Separate databases or schemas per tenant is a real option with real advantages, but it trades a data-access problem for an operations problem, and for a product that expects hundreds of small tenants the migration cost of "run this DDL across four hundred schemas" is not worth it.

Option one: filter in the application

Every query includes WHERE tenant_id = $current_tenant. The application knows who is asking and scopes accordingly. This is what most teams do, usually without deciding to.

Its advantages are real. It is obvious — the filter is visible in the query, so anyone reading the code can see the isolation. It is simple to debug, because a query in a log is the query that ran. It carries no database configuration and no per-connection setup. And the query planner sees the predicate directly, so index selection is straightforward.

The failure mode: it relies on nobody forgetting

Isolation lives in every single query. There is no mechanism preventing a query without the filter — only convention and review. And the ways to forget are mundane:

  • A new endpoint written in a hurry, copying a query from somewhere the scope was already applied upstream.
  • A join to a table whose alias was scoped, while the joined table's own tenant_id is left unconstrained.
  • A background job or scheduled report that has no request context to derive the tenant from, so someone omits it "for now".
  • An admin or support tool that legitimately reads across tenants, later reused as a helper by code that should not.
  • A raw SQL escape hatch for a query the ORM handled badly, which bypasses the repository layer where the scoping was centralised.

The failure is silent. A missing filter returns more rows, not an error. Tests written against a database with one tenant in it pass perfectly, which is why this bug so reliably reaches production: the test fixture is the thing hiding it.

The cheapest thing you can do today: seed at least two tenants in every test database, and assert that queries return only the first one's rows. A large share of cross-tenant leaks are caught immediately by fixtures that contain a second tenant, and never caught at all by fixtures that do not.

Option two: row-level security in Postgres

Postgres can apply the filter itself. Enable RLS on a table, write a policy, and the database appends the predicate to every query against that table — whether or not the application remembered to. 

ALTER TABLE students ENABLE ROW LEVEL SECURITY;

-- Without FORCE, the table owner bypasses its own policies.
-- If your app connects as the owner, RLS silently does nothing.
ALTER TABLE students FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON students
  USING      (tenant_id = current_setting('app.tenant_id')::uuid)
  WITH CHECK (tenant_id = current_setting('app.tenant_id')::uuid);

-- USING filters what you can SEE.
-- WITH CHECK constrains what you can WRITE. Omit it and a tenant
-- can INSERT rows belonging to somebody else.

The application then sets the tenant once per connection checkout, and every query in that transaction is scoped automatically: 

-- set_config(..., true) scopes the setting to the transaction, so a
-- pooled connection cannot carry one tenant's context into the next
-- request. Getting this wrong is the classic RLS incident.
SELECT set_config('app.tenant_id', $1, true);

The failure modes: configuration, pooling and plans

  • It is off by default, per table. A migration adds a table and nobody enables RLS on it. That table now has no protection at all, and the system looks exactly as safe as it did before. This needs a test that enumerates tables and asserts every tenant-scoped one has RLS forced.
  • Superusers and owners bypass policies. Connect as the table owner without FORCE and the policies are inert. Your isolation is a configuration detail of the connection string.
  • Connection pooling is where it goes wrong. A session-level setting on a pooled connection outlives the request that set it. Transaction-scoped settings and per-transaction pooling avoid this; the combination of session-scoped settings and a transaction-mode pooler produces exactly the leak RLS was supposed to prevent.
  • Debugging gets less direct. The query in your log is not the query that ran. Understanding a plan means remembering the invisible predicate is there.
  • Policies interact with plans. A policy using a function that the planner treats as opaque can change how predicates are pushed down. Usually fine, occasionally surprising, and worth checking with EXPLAIN on your hottest queries.

Why both, and what each is for

The two approaches fail in different directions, which is precisely why running both is not redundant.

Application-level scoping is the primary mechanism. It is explicit, it is visible in code review, it produces good query plans, and it makes the intent of each query legible to the next person. Nearly all of the time, it is what is actually doing the work.

RLS is the backstop. It exists for the day the primary mechanism is forgotten. On that day the query returns nothing instead of everything, and you get a confusing empty list and a bug report — rather than a breach and a disclosure obligation. The whole value is in the direction of the failure.

A backstop you never notice is a backstop that is working. The measure of this design is not how often RLS catches something — it is that when it does, the outcome is an empty screen instead of a phone call.

What this costs

Honestly: some setup and some care. A connection-acquisition hook that sets the tenant transactionally. A migration checklist item so new tables get policies. A test asserting every tenant-scoped table has RLS forced. A deliberately privileged role for the small number of genuinely cross-tenant operations, used nowhere else. That is a day of work, most of which you only do once.

What it does not cost is developer ergonomics. Day to day, nobody writing a feature thinks about RLS at all. They write the scoped query as they always would, and the second layer sits underneath doing nothing visible.

What to take away

  • Pick your isolation strategy by asking how each one fails, not which is cleaner. Application filtering fails open; RLS fails closed.
  • Application-level scoping as the primary mechanism, RLS as the backstop, gives you readable queries and a safe failure mode at the same time.
  • FORCE ROW LEVEL SECURITY, always, and check which role your application connects as.
  • Use WITH CHECK, not just USING — otherwise you have filtered reads and unfiltered writes.
  • Set tenant context per transaction, not per session, and understand your pooler's mode before you trust it.
  • Put two tenants in every test fixture. It is the highest-value hour in this entire article.
