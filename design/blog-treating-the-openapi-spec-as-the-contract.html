Treating the OpenAPI spec as the contract, not the documentation

Most teams write the API, then write a spec describing it, then watch the two drift apart over the next six months. Generate the client types from the spec instead, and fail CI when the implementation no longer matches it, and the drift becomes a build error rather than a production bug. The frontend and backend stopped disagreeing the week we turned this on.

There is a specific class of bug that only exists on teams with more than one person. The backend renames a field from customer_name to customerName. The change is correct, the tests pass, the endpoint returns 200. Two days later someone reports that a name is blank on a screen nobody opened during QA. The frontend was reading a key that no longer exists, and JavaScript has no opinion about that: it just hands back undefined and renders nothing.

Nothing caught it because nothing was watching the boundary. Both sides had tests. Both sides were green. The tests just never asked each other what they expected.

Documentation drifts; contracts don't

The distinction is about direction. A spec written after the implementation is documentation: a description that was accurate once. Nothing depends on it, so nothing notices when it rots. Within a couple of months it is a file people stop opening, because it has been wrong before and they no longer trust it.

A spec that the build depends on is a contract. If it is wrong, something breaks immediately and loudly. That is the entire trick — not better discipline, just moving the spec somewhere that discipline is enforced mechanically.

Spec-first or code-first, either is fine

There is a long-running argument about whether to hand-write the OpenAPI document and generate server stubs from it, or to write the code with annotations and generate the document. Both work. What matters is that exactly one of them is the source of truth and the other is generated, never edited by hand, and checked in.

We usually go code-first, because keeping the annotations next to the handler makes them far more likely to be updated in the same commit as the behaviour. The generated document is committed to the repository, and CI regenerates it and fails if the committed copy differs. 

# The whole gate, in four commands.

pnpm openapi:generate      # code -> openapi.json
git diff --exit-code openapi.json
   # ^ fails if the committed spec is stale

pnpm openapi:lint          # spec is valid + follows house rules
pnpm types:generate        # openapi.json -> api-types.d.ts
pnpm typecheck             # frontend must satisfy those types

Rename a field in a handler and forget to update the frontend, and the failure now happens at step five, on your machine, before the pull request exists. The bug is not caught earlier — it is never created.

What the generated types actually buy you

Generating TypeScript from the spec is the part that changes day-to-day work. A hand-written interface Booking in the frontend is a guess about the API that happened to be right on the day it was written. A generated one cannot be a guess.

  • Renamed and removed fields become compile errors. The class of bug in the opening of this post stops being possible.
  • Nullability stops being folklore. If the spec says a field is optional, the generated type says string | undefined, and every call site is forced to handle the missing case. Teams are consistently wrong about which fields can be null; the spec is the only place to settle it once.
  • Enums stay in sync. Add a pending_payment status on the backend and the frontend's exhaustive switch immediately fails to compile, pointing at every screen that needs a decision about it.
  • Request shapes are checked too. Not just responses — the body you send, the path parameters, the query string.
  • The spec becomes worth reading. Because it is load-bearing, it is accurate; because it is accurate, people use it instead of asking in chat.
Generate types, not clients. Full client generators produce a lot of code you did not write, with their own opinions about fetching, retries, caching and error handling — and you will end up fighting them the first time you need a request cancelled or a response cached. Generating only the types and keeping a small hand-written fetch wrapper gives you the safety without inheriting a framework.

Validate the implementation, not just the document

A spec can be perfectly valid and still describe something the server does not do. Schema linting proves the document is well-formed; it proves nothing about the running code.

Assert against the spec in integration tests

The cheap version of contract testing is to load the spec in the API test suite and assert that every real response validates against the schema for that path and status code. It is a handful of lines of setup and it catches the cases annotations miss — a field the serialiser drops, a date formatted differently than declared, an error shape that only appears on the 409 path. 

// One assertion, applied to every response the suite produces.
const res = await api.post('/bookings', payload);

expect(res.status).toBe(201);
expectValidAgainstSpec(res, {
  path:   '/bookings',
  method: 'post',
  status: 201,
});

// Also assert the FAILURE paths. Error responses are the ones that
// drift most, because nobody looks at them until production does.
const conflict = await api.post('/bookings', overlapping);
expect(conflict.status).toBe(409);
expectValidAgainstSpec(conflict, {
  path: '/bookings', method: 'post', status: 409,
});

Shared models via $ref, or the duplication comes back

A spec with Booking defined inline in six places is not a single source of truth — it is six sources that currently agree. Every shared shape belongs in components/schemas and is referenced with $ref everywhere it appears. This is the same rule as not copy-pasting a type definition in application code, and it is broken just as often, because inline schemas are what most annotation tools produce by default.

The failure modes worth knowing

  • A spec nobody generates from. If the frontend still hand-writes its types, you have documentation again, with extra steps.
  • Committed artefacts nobody regenerates. Without the git diff --exit-code step, the committed spec quietly falls behind the code and the generated types describe last month's API very confidently.
  • Overly permissive schemas. additionalProperties: true and untyped object fields make validation pass while proving nothing. If a schema accepts anything, it is not a contract.
  • Treating any change as a breaking change. Adding an optional response field is safe; removing a field or tightening a type is not. Knowing the difference is what keeps the process from becoming a tax on every pull request.

What to take away

None of this is about OpenAPI specifically. The principle is that an agreement between two systems should be represented once, in a form that both sides derive from, with a build step that fails when they diverge. OpenAPI is just the most widely supported way to write that agreement down for an HTTP API.

The three things that make it work, in order of importance: generate the client types rather than writing them; commit the spec and fail CI when it is stale; validate real responses against the schema, including the error paths. Skip any one of them and you are back to a document that describes an API you hope you still have.
